# Residual MLP Fixture Contract

This contract defines the next bounded fixture after
`token_loop_attention_fixture`.

The parent agent owns this contract. HDL sub-agents implement RTL or RTL
generator changes. The parent agent must not hand-write generated HDL.

## Scope

Create a small deterministic fixture that proves residual add plus a
LLaMA-style MLP/SwiGLU scheduling path can be represented as composable
SystemVerilog.

This milestone is a fixture, not a target-scale LLaMA MLP. It should prove the
control/data contract for:

- residual add after an attention-like output;
- MLP norm input selection from the residual stream;
- gate/up/down projection scheduling metadata;
- deterministic SwiGLU-style activation arithmetic on a tiny vector;
- final residual add after the MLP path.

It does not implement target-scale projection streaming, DDR/AXI movement,
full SwiGLU numerical fidelity, full hidden/intermediate dimensions, complete
decoder layer execution, or board-level signoff.

## Kernel Name

Use:

- CLI kernel: `residual_mlp_fixture`
- HDL module: `residual_mlp_fixture`
- report artifact: `residual_mlp_fixture_golden.json` plus
  `kernel_report.json`

## Required Interface

Follow `docs/hdl_module_interface_contract.md`.

Common control:

- `input logic aclk`
- `input logic aresetn`
- `input logic start_i`
- `output logic done_o`

Top-level data should stay compact:

- small registered hidden input vector;
- small registered attention output vector;
- small registered final output vector;
- compact status/trace summary.

Top-level data must not expose:

- full hidden-size vectors;
- full intermediate-size MLP tensors;
- full gate/up/down weight matrices;
- full activation, scale, zero-point, or debug trace arrays;
- 128-bit memory response words as package-level ports.

## Deterministic Fixture Math

Use a fixed tiny vector width, preferably 4 elements, with signed integer
arithmetic. The sub-agent may choose exact vectors, but must write them into
`residual_mlp_fixture_golden.json` and reproduce them in the report.

Required dataflow:

1. `residual0 = hidden_input + attention_output`.
2. `mlp_norm_input = residual0`.
3. Produce tiny deterministic `gate_proj` and `up_proj` vectors.
4. Apply a documented bounded SwiGLU-style policy, for example
   `silu_approx_gate_times_up_shift`, and record the policy.
5. Produce a tiny deterministic `down_proj` vector.
6. `final_output = residual0 + down_proj`.

The fixture may use small constant matrices or internally generated projection
metadata. If constants are used, they must be reported and compared against a
Python/NumPy golden reference. If child projection fixtures are instantiated,
their handshake evidence must be dynamically observed and reported.

## Sequencing Requirements

The residual/MLP FSM must:

1. Accept `start_i` while idle.
2. Execute residual add 0.
3. Execute or schedule MLP norm/input staging.
4. Execute or schedule gate/up projection fixture stage.
5. Execute bounded SwiGLU-style activation.
6. Execute or schedule down projection fixture stage.
7. Execute final residual add.
8. Latch final output/status and assert `done_o`.
9. Keep `done_o` asserted until `start_i` is deasserted.

If any child fixture is called, hold child `start_i` while busy, observe child
`done_o`, deassert child `start_i`, and wait for child `done_o` to clear before
the next child call or repeated call.

## Dynamic Evidence Requirements

Simulation and report evidence must include:

- ordered FSM trace with these semantic events:
  `residual0_start`, `residual0_done`, `gate_up_start`, `gate_up_done`,
  `swiglu_start`, `swiglu_done`, `down_start`, `down_done`,
  `residual1_start`, `residual1_done`;
- hidden input vector and attention input vector;
- observed `residual0` vector;
- observed gate projection vector and up projection vector;
- observed SwiGLU/intermediate vector and documented approximation policy;
- observed down projection vector;
- final output vector;
- output/status stability while `done_o` is high;
- compact I/O evidence.

If gate/up/down are only deterministic fixture constants, report them as
fixture constants and do not label them as dynamically streamed projection
evidence. If they are generated by child projection fixtures, the testbench
must read child ports or hierarchy and compare observed values.

## Required Report Fields

`kernel_report.json` must include:

- `kernel: residual_mlp_fixture`;
- `coverage_level: residual_mlp_fixture`;
- `residual_mlp_fixture: true`;
- `fixture_hidden_width` and `fixture_intermediate_width`;
- FSM state order;
- ordered residual/MLP trace from simulation;
- hidden input, attention input, `residual0`, gate, up, SwiGLU, down, and final
  output vectors;
- arithmetic policy, including signed width, shifts, saturation or clipping,
  and whether true SiLU/exponential math is implemented in RTL;
- whether gate/up/down projections are fixture constants or child projection
  fixtures;
- child module list if any children are instantiated;
- top-level interface summary and exposed port width summary;
- `implementation_stage`, preferably `post-route`;
- Verilator evidence when enabled;
- Vivado timing/resource evidence when synthesis is enabled;
- timing caveat that routed timing is fixture timing without board-level I/O
  delay, DDR/AXI shell, or PS/PL integration constraints;
- `omitted_operations`;
- `does_not_claim`.

## Required Omitted Operations

Unless independently implemented and verified, the report must list these as
omitted or not claimed:

- target-scale LLaMA MLP dimensions;
- GPTQ INT4 packed gate/up/down weight streaming;
- DDR/AXI packed-weight movement;
- true SiLU or exponential activation math in RTL;
- full hidden-size RMSNorm before MLP;
- target multi-layer LLaMA iteration;
- token-dependent full-sequence KV-cache accumulation;
- logits generation and sampling;
- full LLaMA layer execution;
- full LLaMA model execution;
- board shell and PS/PL integration;
- board-level signoff.

## Pass Criteria

- Unit tests pass.
- RTL simulation passes.
- The common handshake contract is met.
- Ordered residual/MLP trace is dynamically observed and checked.
- Numeric output matches Python/NumPy golden vectors.
- Outputs are stable while `done_o` is high.
- Report evidence labels distinguish fixture constants from dynamically
  observed child projection evidence.
- Verilator evidence is recorded and passes when enabled.
- Vivado timing has non-`NA` setup, hold, and pulse-width checks and zero
  failing endpoints when synthesis is enabled.
- Report text does not claim target-scale MLP, DDR/AXI movement, full LLaMA
  layer/model execution, or board-level signoff.
- A read-only verification agent finds no P0/P1 issues.
